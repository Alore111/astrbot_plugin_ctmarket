# Trae Rules（本仓库）

## 总体目标

- 代码优先可读、可维护；其次性能；最后再做“炫技式抽象”。
- 插件属于消息监听高频路径：不要在监听器里做阻塞 IO、不要刷屏日志。
- 数据持久化遵循 AstrBot 规范：大文件/数据库放到 data/plugin_data/{plugin_name}（优先用 StarTools 获取数据目录）。

## Python 代码规范

- 文件职责单一：规则匹配、存储、Web API、前端页面彼此解耦。
- 命名清晰：用完整单词表达含义，避免缩写（q/ts 等行业常用缩写除外）。
- 类型提示：对模块边界与公共函数增加类型提示；对难理解的数据结构加 dataclass。
- 异步安全：任何可能阻塞的操作（SQLite、文件 IO、网络）必须放到 asyncio.to_thread 或异步实现中。

## 注释与文档字符串

- 只在“读代码无法直观看出原因/约束”的地方写注释：
  - 为什么要缓存规则、为什么要使用 WAL、为什么要避免在监听器里打印内容等
- 公共函数/类必须写 docstring，说明：
  - 输入/输出是什么
  - 关键约束（比如 include_* 为空的语义）
  - 重要的副作用（比如会写数据库）
- 禁止“复述代码”的注释（例如 `i += 1  # i 加 1`）。

## 日志规范（astrbot.api.logger）

- 统一前缀：`[{PLUGIN_NAME}]`。
- 统一风格：`event=xxx key=value key=value`（方便 grep 与结构化检索）。
- 日志级别约定：
  - info：生命周期与一次性事件（启动、配置加载、规则刷新、存储准备完成）
  - warning：可恢复问题（无效正则、配置缺失但有默认值兜底）
  - exception：不可预期异常（写库失败、Web API 异常），必须带上下文（group_id/message_id 等）
  - debug：高频事件（逐条消息处理、分页请求）默认不开启，避免刷屏
- 敏感/高体量数据禁止直接写日志：
  - message_chain/raw_message/完整 message_str
  - 仅在必要时记录 ID、长度、摘要（例如 `text_len=...`），避免泄露群聊内容。

## 错误处理

- 监听器内任何异常必须被捕获并记录 exception，避免影响其他消息处理。
- Web API 参数解析必须容错（非法数字回退默认值），返回值结构保持稳定。

## 前端（插件 Pages）规范

- 必须使用 `window.AstrBotPluginPage` bridge 调用后端，不直接拼接 Dashboard 的鉴权/同源能力。
- 页面脚本尽量保持无框架、少依赖；交互逻辑集中在 app.js。
- 出错要可见：请求失败时在页面上渲染错误信息，不静默失败。
